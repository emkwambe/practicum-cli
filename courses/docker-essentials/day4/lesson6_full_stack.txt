# Lesson: A Real Compose Stack — Web + API + Database + Cache

## [WHEN] You need this when:
1. You're ready to put it all together.
2. You need a reference for "real" production-shaped compose files.
3. You're onboarding a new dev to your project.

## [INTENT] "Set up a complete development stack with one command."

## [CONTEXT]
This lesson combines everything from Days 1-4 into a real-world stack:
  - PostgreSQL database (persistent storage)
  - Redis cache
  - Backend API (built from your code)
  - Frontend web server
  - Multi-network isolation
  - Health checks
  - Restart policies
  - Environment variables from .env

## [ACTION] Production-shaped compose.yml

  # docker-compose.yml
  services:
    db:
      image: postgres:16-alpine
      environment:
        POSTGRES_DB: ${DB_NAME:-myapp}
        POSTGRES_USER: ${DB_USER:-app}
        POSTGRES_PASSWORD: ${DB_PASSWORD:?password required}
      volumes:
        - dbdata:/var/lib/postgresql/data
        - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql:ro
      networks:
        - backend
      healthcheck:
        test: ["CMD-SHELL", "pg_isready -U ${DB_USER:-app}"]
        interval: 10s
        timeout: 5s
        retries: 5
      restart: unless-stopped

    cache:
      image: redis:7-alpine
      networks:
        - backend
      restart: unless-stopped
      healthcheck:
        test: ["CMD", "redis-cli", "ping"]
        interval: 10s

    api:
      build:
        context: ./backend
        dockerfile: Dockerfile
      environment:
        DATABASE_URL: postgres://${DB_USER:-app}:${DB_PASSWORD}@db:5432/${DB_NAME:-myapp}
        REDIS_URL: redis://cache:6379
        NODE_ENV: ${NODE_ENV:-development}
      depends_on:
        db:
          condition: service_healthy
        cache:
          condition: service_healthy
      networks:
        - backend
        - frontend
      healthcheck:
        test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
        interval: 30s
        retries: 3
      restart: unless-stopped

    web:
      build:
        context: ./frontend
        dockerfile: Dockerfile
      environment:
        API_URL: http://api:3000
      depends_on:
        api:
          condition: service_healthy
      ports:
        - "${WEB_PORT:-8080}:80"
      networks:
        - frontend
      restart: unless-stopped

  volumes:
    dbdata:

  networks:
    backend:
      internal: true       # backend has no internet access
    frontend:

  # .env file (DO NOT commit to git):
  #   DB_PASSWORD=changeme
  #   WEB_PORT=8080
  #   NODE_ENV=production

## [EXPECTED RESULT]
With one command, your entire stack starts in dependency order:
  1. db starts → becomes healthy
  2. cache starts → becomes healthy
  3. api starts (waits for db + cache) → becomes healthy
  4. web starts (waits for api)

  docker compose up -d
  docker compose ps             # all "healthy"
  curl http://localhost:8080    # works!

## Try it now:
Adapt this to your real project structure. Replace ./backend and ./frontend
with your actual code directories. Add a .env file with secrets.

Test the full lifecycle:
  docker compose up -d
  docker compose ps
  docker compose logs -f api
  docker compose down
  docker compose down -v        # full reset

## Technician Tip:
For multiple environments, use override files:
  docker-compose.yml             # base config (committed to git)
  docker-compose.override.yml    # dev overrides (auto-loaded, in .gitignore)
  docker-compose.prod.yml        # prod overrides (used in CI)

  # Local dev:
  docker compose up -d           # auto-applies override.yml

  # Production:
  docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d

## Common Pain Point:
"depends_on with condition: service_healthy isn't waiting" → the upstream
service doesn't have a HEALTHCHECK defined. Add one or downgrade to
condition: service_started.
