# Lesson: docker build — Turn Dockerfile into an Image

## [WHEN] You need this when:
1. You wrote a Dockerfile and need to produce an image from it.
2. You changed your app code and need to rebuild.
3. You want to tag an image for a specific version.
4. Your CI/CD pipeline builds images on every commit.

## [INTENT] "Build an image from my Dockerfile."

## [CONTEXT]
docker build reads your Dockerfile and produces an image.
The image is stored locally. Tag it so you can reference it by name
(instead of a hash).

The "build context" is the directory Docker sends to the daemon.
Everything in it is accessible to COPY — so keep it small.
Use a .dockerignore file (like .gitignore) to exclude unnecessary files.

## [ACTION] docker build [options] <context>
Common patterns:
  docker build -t myapp .                    — build from current dir, tag "myapp"
  docker build -t myapp:v1.0 .               — semantic version tag
  docker build -t myapp:v1.0 -t myapp:latest .  — multiple tags
  docker build --no-cache -t myapp .         — force rebuild from scratch
  docker build -f Dockerfile.prod -t myapp . — use specific Dockerfile
  docker build --build-arg VERSION=1.0 .     — pass a build variable
  docker build --progress=plain -t myapp .   — see full output (no truncation)

## [EXPECTED RESULT]
Docker prints each layer as it builds. If a step fails, the build
stops and shows which step and why.
"Successfully tagged myapp:latest" means success.
Run docker images to see your new image.

## Try it now:
  cd my-first-image   # from previous lesson
  docker build -t hello:v1 .
  docker images | grep hello
  docker run --rm hello:v1

## Technician Tip — .dockerignore:
Create a .dockerignore file to exclude noise:

  .git
  node_modules
  *.log
  .env
  __pycache__
  .DS_Store

This makes builds FASTER and avoids leaking secrets into images.

## Common Pain Point:
"COPY failed: no source files" → file not in build context,
OR excluded by .dockerignore, OR wrong path in Dockerfile.

"Build is slow every time" → your COPY layer probably invalidates
the cache. Reorder so dependency files (requirements.txt, package.json)
are copied BEFORE application code.
