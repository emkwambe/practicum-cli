# Lesson: Secrets — Don't Put Passwords in Environment Variables

## [WHEN] You need this when:
1. Your app needs database passwords, API keys, or TLS certificates.
2. Security team forbids env vars for credentials.
3. You're moving from local dev to production.
4. You need to rotate secrets without rebuilding images.

## [INTENT] "Provide credentials to my container without exposing them."

## [CONTEXT]
Environment variables for secrets are PROBLEMATIC:
  - Visible in: docker inspect, ps aux, /proc/<pid>/environ
  - Get logged when apps print env (debug mode, error pages)
  - Persist in image history if set with ENV in Dockerfile
  - Can leak in stack traces

Better options (in order of preference):
  1. Docker Swarm secrets (best for multi-node)
  2. Kubernetes secrets (in Kubernetes)
  3. External secret managers (Vault, AWS Secrets Manager, GCP Secret Manager)
  4. Bind-mounted files (acceptable for single-host)
  5. Environment variables (last resort, dev only)

## [ACTION] Use secrets safely

### Option 1: Docker Swarm secrets (covered Day 8)
  echo "supersecret" | docker secret create db_pass -
  docker service create \
    --name api \
    --secret db_pass \
    -e DB_PASS_FILE=/run/secrets/db_pass \
    myapp

  # Inside container, app reads /run/secrets/db_pass
  # File only visible to that container's processes.

### Option 2: Bind-mount a secret file (single host)
  # Create the secret file with restrictive permissions:
  echo "supersecret" > /etc/myapp/db_pass
  chmod 600 /etc/myapp/db_pass

  docker run -d \
    --name api \
    -v /etc/myapp/db_pass:/run/secrets/db_pass:ro \
    -e DB_PASS_FILE=/run/secrets/db_pass \
    myapp

  # App reads file: open('/run/secrets/db_pass').read().strip()

### Option 3: Compose secrets (docker-compose v3.1+)
  services:
    api:
      image: myapp
      secrets:
        - db_pass
      environment:
        DB_PASS_FILE: /run/secrets/db_pass

  secrets:
    db_pass:
      file: ./secrets/db_pass.txt    # local file
      # OR: external: true            # use existing Swarm secret

### Option 4: External secret manager (production gold standard)
  # Vault example (sketch):
  docker run -d \
    -e VAULT_ADDR=https://vault.example.com \
    -e VAULT_TOKEN=$VAULT_TOKEN \
    myapp:vault-aware

  # Inside the app, fetch secrets via Vault API at startup.

### Pattern: read secret from FILE, fall back to env
  # Useful for transition or supporting multiple environments
  password = (
      open('/run/secrets/db_pass').read().strip()
      if os.path.exists('/run/secrets/db_pass')
      else os.environ.get('DB_PASS')
  )

## [EXPECTED RESULT]
Secrets never appear in:
  - docker inspect output
  - process environment listing
  - image layers
  - shell history
  - logs

## Try it now:
  echo "my-secret-password" > /tmp/db_pass
  chmod 600 /tmp/db_pass

  docker run --rm \
    -v /tmp/db_pass:/run/secrets/db_pass:ro \
    alpine cat /run/secrets/db_pass

  rm /tmp/db_pass

## Technician Tip — image scrubbing:
If you accidentally baked a secret into an image:
  1. Rotate the secret IMMEDIATELY (assume it's compromised)
  2. Delete the image from all registries
  3. Build a new image WITHOUT the secret
  4. Update all running containers

You CANNOT just "docker rmi" — the secret is in the image layers
forever, including any pulled copies.

## Common Pain Point:
"I put secrets in ENV in my Dockerfile" → those are part of the image
layer history. Rebuild without them and rotate the credentials.

"My CI/CD pipeline has secrets in plain text" → use the CI's secret
management (GitHub Secrets, GitLab CI variables, etc.). Never commit.

## DCA Exam Note:
Know that Swarm secrets are MOUNTED at /run/secrets/<name> (tmpfs, in memory).
Know that secrets are encrypted at rest in the Swarm Raft store.
Know that env vars are visible via docker inspect; secrets are not.
Know that secrets work only in Swarm services, not standalone containers.
